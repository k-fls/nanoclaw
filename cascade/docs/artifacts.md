# Persistent surface

Everything not listed here is derived from git at runtime.

## Committed files

| Path | Role | Who writes | Who reads |
|---|---|---|---|
| `.cascade/branch-classes.yaml` | Branch class patterns and metadata | humans (rare) | all scripts |
| `.cascade/config.yaml` | Repo-wide knobs (version_depth, upstream remote) | humans (once) | all scripts |
| `.cascade/ownership_rules` | gitignore-style patterns; `?` prefix marks safety-net | humans (rare) | `check.ts`, ownership derivation |
| `.cascade/ownership_overrides` | Explicit `path  owner` escape hatch for ambiguous history | humans (rare) | ownership derivation, `check.ts` |
| `.cascade/parent_branch` | Version source for an edition; only on branches that need it | humans (per edition) | `version.ts`, `check.ts` |
| `.cascade/bypass-log` | Append-only record of acknowledged CI bypasses; supports `upstream/*` policy entries | `cascade bypass` (humans invoke) | `check.ts` |
| `Cascade-Hotfix-Pair:` commit trailer | Symmetric pair on cherry-pick + `core`-merge commits; closes the hotfix loop. Script-written, not an author convention. | `cascade hotfix` | `check.ts` (`hotfix-loop-open`) |

| Path | Regenerated by | Gitignored |
|---|---|---|
| `/.ownership_map.txt` | `cascade ownership` | yes |

## File schemas

### `.cascade/branch-classes.yaml`

```yaml
classes:
  - name: upstream
    pattern: '^upstream/.+$'
    read_only: true

  - name: core
    pattern: '^core$'
    base: upstream/main
    version_source: upstream/main

  - name: module
    pattern: '^module/[^/]+$'
    base: core
    version_source: core

  - name: channel
    pattern: '^channel/[^/]+$'
    base: core
    version_source: core
    co_merges: ['upstream/skill/*']

  - name: skill
    pattern: '^skill/[^/]+$'
    base: core
    version_source: core

  - name: skill-adapter
    pattern: '^skill/([^/]+)/[^/]+$'
    base_from_match: 'skill/$1'
    version_source_from_match: 'skill/$1'

  - name: module-adapter
    pattern: '^module/([^/]+)/[^/]+$'
    base_from_match: 'module/$1'
    version_source_from_match: 'module/$1'

  - name: edition
    pattern: '^edition/[^/]+$'
    base: core
    version_source: declared  # via .cascade/parent_branch

  - name: deploy
    pattern: '^deploy/[^/]+$'
    base: 'edition/*'
    version_source_from_ancestry: 'edition/*'

  - name: ephemeral
    pattern: '.*'
    fallback: true
    not_versioned: true
```

Topological order for ownership derivation follows the list order (first match wins).

### Ephemeral contract

- **Ownership.** Ephemerals are never candidate owners (see [ownership.md](ownership.md)). A file whose introducing commit lives on a merged-and-deleted ephemeral is owned by the first long-lived branch in whose ancestry the commit lives.
- **Versioning.** Ephemerals are `not_versioned: true`. `versionSourceOf(<ephemeral>)` returns `null`; callers that reach for a version on an ephemeral get an actionable error rather than a derived guess. Phase 2's `version.ts` mutating is the first concrete consumer of this rule.
- **Naming conventions** (checked by `check.ts`'s ephemeral-fallback class):
  - `hotfix/<slug>` — reserved for the `cascade hotfix` two-target flow. `hotfix.ts` creates, tracks, and (after step 3) deletes these branches.
  - All other ephemeral names are operator-chosen; no further structure required.

### `.cascade/parent_branch`

One line, the branch name of the version source:

```
core
```

Present only on `edition/<name>` branches (or on any other branch that needs to override the mechanical default).

### `.cascade/bypass-log`

Append-only. Columns: commit (SHA or policy pattern), date, branch, bypassed rule, reason.

**Rule column is closed-set.** Valid values are the rule names `check.ts` actually emits; arbitrary strings aren't accepted (a typo that matches nothing would silently fail to suppress anything). Currently emitted (Phase 0): `double-introduction`, `merge-preserve`, `base-validity`, `prefix-mismatch`, `override-invalid`, `override-redundant`, `dead-rule`, `hygiene`, `unowned-new-file`. Later phases add their own; `check.ts` validates the column against its own rule registry at run time.

```
abc1234     2026-04-13  deploy/prod-acme  merge-preserve       hotfix during P1 outage; followup core-hotfix-2026-04-13
def5678     2026-04-20  edition/starter   prefix-mismatch      intentional cross-version test edition
upstream/*  2026-04-17  main              double-introduction  upstream doesn't follow cascade rules
upstream/*  2026-04-17  main              merge-preserve       upstream uses its own merge policy
```

`check.ts` consults this to suppress warnings on listed commits. New violations not in the log still fail.

**Policy patterns.** Entries whose commit field is `upstream/*` are standing policies: they match any violation whose attached commit is reachable from the upstream ref declared in `config.yaml`. The upstream-reachable commit set is recomputed on every run, so policy entries cover future upstream commits automatically — the date is the acknowledgement timestamp, not a cutoff. A violation with multiple commits (e.g. double-introduction) is suppressed if *any* of its commits is upstream-reachable, so intake artifacts (the target-side "merged upstream" commits paired with upstream ones) are covered alongside pure-upstream violations.

**Accepted risk: bypass-log is trust-based.** Any committer can append an entry and suppress the corresponding warning. This is intentional — the threat model here is accidental-mistake, not adversarial-dev. Forge-level gates on bypass-log changes are disproportionate to the risk given how many other things a committer can break.

### `.cascade/ownership_overrides`

Explicit path → owner mapping. The escape hatch for files whose history is genuinely ambiguous (pre-cascade squash/cherry-pick duplication, intentional cross-branch re-authoring, etc.). Overrides are consulted *before* mechanical derivation.

Format: one entry per line, `path  owner-branch`, whitespace-separated. Lines beginning with `#` are comments.

```
# Pre-cascade double introductions: legitimately core files even though
# derivation finds matches on a descendant branch first.
src/dir-fingerprint.test.ts    main
src/interaction/session.ts     main
src/mount-security.test.ts     main
src/test-helpers.ts            main
```

Effects:
- Derivation returns the declared owner instead of the first-parent/ancestry match.
- Double-introduction warning is suppressed for the overridden path (the human has explicitly acknowledged the history).
- `check.ts` flags overrides whose owner isn't a long-lived branch (`override-invalid`, error) and overrides that duplicate what derivation would produce anyway (`override-redundant`, info) — so the file doesn't rot.

**Not a central ownership registry.** Files in the repo are still owned via the derivation algorithm (§9); overrides address the narrow cases where derivation has no mechanically correct answer. If the list grows beyond a handful of entries, it's a signal that the derivation algorithm or the branch model needs attention, not that more overrides should be added.

### Edition snapshot (embedded in tag body)

```json
{
  "schema_version": 1,
  "edition": "starter",
  "version": "1.9.0.2",
  "generated_at": "2026-04-13T10:00:00Z",
  "core_version": "1.9.0.5",
  "upstream_version": "1.9.0",
  "included": {
    "channels": ["whatsapp", "telegram"],
    "skills": ["image-vision"],
    "adapters": [{"skill": "reactions", "channel": "whatsapp"}]
  },
  "ownership_map": {
    "src/channels/telegram/sender.ts": "channel/telegram",
    "src/index.ts": "core"
  },
  "branch_classes": { "...": "copy of .cascade/branch-classes.yaml" }
}
```

Lets a downstream deploy run P3 locally without the full branch graph. Embedded in the annotated-tag body of every `edition/<name>/<A.B.C.D>` tag — see [§ Tag body schema](#tag-body-schema). Snapshot computation: see `cascade/scripts/edition-snapshot.ts`; runs in-memory at tag time, nothing written to the working tree.

**Schema versioning.**

- `schema_version` is an integer at the top level. Phase 2 ships `1`.
- **Compatibility matrix** (tag's `schema_version` vs. consumer CLI's `MAX_SNAPSHOT_SCHEMA`):

  | Tag vs. MAX | Result |
  |---|---|
  | `tag < MAX` | read through |
  | `tag = MAX` | read |
  | `tag > MAX` | refuse with `unsupported-snapshot-version` |

  No silent partial reads in any direction.
- Additive changes (new fields) leave `schema_version` unchanged. Older consumers read fewer fields, never wrong fields.
- Breaking changes (renamed / removed fields, semantic shifts) bump `schema_version`.
- Max supported schema version per CLI is a source constant in `cascade/scripts/snapshot-schema.ts` (`MAX_SNAPSHOT_SCHEMA`). Kept out of `package.json` so the publish manifest stays free of internal capability metadata.
- Consumer newer than tag (tag's `schema_version` ≤ CLI's max): read as-is. Within a single CLI major, all prior schemas are supported. A major CLI bump *may* drop support for older schemas; downstream repos update their `cascade/` submodule before consuming editions written by the new major.
- Consumer older than tag (tag's `schema_version` > CLI's max): `cascade snapshot` refuses with `unsupported-snapshot-version`.

The `---cascade-snapshot---` fence text itself is not versioned — changing the fence would break older consumers' ability to even locate the JSON. Evolution happens inside the JSON, never at the fence.

## Tag body schema

All cascade-written tags go through the single `writeTag()` helper in `version.ts`. Interface:

```ts
writeTag(args: {
  branch: string;          // e.g. 'edition/starter'
  version: Version;        // {A, B, C, D}
  notes?: string;          // human release notes; empty allowed
  snapshot?: Snapshot;     // required iff branch matches edition/*
}): { tag: string; sha: string };
```

Contract:

- **Tag name.** `<branch>/<A.B.C.D>`. Any other shape is rejected.
- **Annotated only.** Lightweight tags are refused.
- **Body composition** (deterministic, single template):

  ```
  <branch> <A.B.C.D>

  <notes or empty line>

  ---cascade-snapshot---
  <snapshot JSON, pretty-printed, trailing newline>
  ---cascade-snapshot-end---
  ```

  The snapshot block is omitted (no fences) when `snapshot` is absent. Fences are grep-friendly; no YAML front-matter, no JSON-in-JSON escaping.

- **Refuse-to-overwrite.** If `<branch>/<A.B.C.D>` already exists in the **local** ref store, the call aborts. Remote collision is not checked — `git push --tags` fails on non-fast-forward tag updates by default, and forge-level branch protection is the authoritative gate. This keeps `writeTag()` networkless.

  Named threats this check defends against (belt + suspenders over the `cascade propagate` step-1 check):

  | Threat | Mitigation |
  |---|---|
  | Concurrent `cascade tag` and `cascade propagate` in two shells | Both funnel through `writeTag()`; whichever reaches it second aborts. |
  | Operator runs `git tag` by hand between step-1 inspect and writeTag call | Step 1 is stale; writeTag catches. |
  | Bug in step-1's own detection logic | writeTag is the independent last line. |

  No `--force`, no bypass-log path. For a pushed tag the only legitimate response is a follow-up bump. For a local-only stale ref, the operator deletes with `git tag -d <tag>` and re-runs — the chokepoint is preserved because writeTag still writes the canonical body.

- **Logging.** Every successful write emits `wrote tag <name> -> <sha>` on stderr.
- **CI parsing.** On any `<branch>/<A.B.C.D>` tag push, CI re-parses the body, validates the fence structure, and (for editions) schema-validates the embedded snapshot. A malformed body never enters the pin graph.

## Halt envelope schema

Structured output format for any `cascade propagate` halt or refusal. Emitted on stdout as JSON (always; the default human renderer is a formatted view of the same object).

```json
{
  "halted": {
    "hop": "core -> channel/telegram",
    "kind": "merge-conflict",
    "details": { "conflicted_files": ["src/channels/telegram/sender.ts"] },
    "remediation": "resolve conflicts, git commit, then re-run cascade propagate"
  },
  "progress": {
    "done": ["upstream -> core"],
    "pending": [
      { "hop": "core -> channel/telegram",              "blocked_by": null },
      { "hop": "channel/telegram -> edition/starter",   "blocked_by": "core -> channel/telegram" },
      { "hop": "edition/starter -> deploy/prod-acme",   "blocked_by": "core -> channel/telegram" },
      { "hop": "core -> channel/whatsapp",              "blocked_by": null }
    ]
  }
}
```

- `halted.hop` is **nullable**: on pre-flight refusals (see halt registry below) no hop has been selected yet, and `halted.hop: null`. Consumers use a single schema for both pre-flight and mid-run.
- `halted.kind` is drawn from the halt registry (below).
- `halted.details` is an open object; shape varies by kind.
- `halted.remediation` is a human-readable one-liner.
- `progress.done` lists hops already completed this invocation.
- `progress.pending.blocked_by` names the halted hop whose resolution the pending hop depends on (null for independent siblings). Agent-facing consumers use this to decide mechanically whether a `cascade propagate --after <branch>` session-scoped skip covers a given pending hop.

On a pre-flight refusal, `progress.done` is empty and `progress.pending` lists the full plan.

## Halt registry

Closed set of halt / refusal kinds raised by Phase 2 commands. All feed the shared rule registry that `bypass-log` entries can reference. Halts are distinct from **CLI usage errors** (invocation mistakes like unrecognized flags or `--after` with no halted hop) — those exit with code 2, print a short message to stderr, emit no halt envelope, and are not bypass-loggable. Usage errors are not part of this registry; they live in each command's argument parser.

Comparison of adjacent kinds:
- **`no-prior-tag` vs. `source-tag-missing`.** `no-prior-tag` fires from `cascade tag` when the branch has no `<branch>/*` tag **and** no resolvable version source — there is no base to even attempt a walk from. `source-tag-missing` fires when the source *is* resolvable but walking its tag history back to the merge-base turns up nothing taggable. Both remediate with `--seed`; the first means "cascade can't find a source," the second means "cascade found the source but it has no tags reachable from here yet."

| Kind | Raised by | When | Remediation |
|---|---|---|---|
| `no-prior-tag` | `cascade tag` | Branch has no prior `<branch>/*` tag and no resolvable version source. | Supply `--seed <A.B.C.D>`. |
| `seed-rejected` | `cascade tag` | `--seed` supplied on a branch where the version is derivable. | Drop `--seed`. |
| `bad-state` | `cascade propagate` (pre-flight) | Working tree dirty at run start — untracked / unstaged / staged changes present, **excluding** `MERGE_HEAD` / unmerged-index state. | Clean or stash; re-run. |
| `remote-missing` | `cascade propagate` (pre-flight, downstream mode only) | Downstream repo has no `source` remote (or configured `downstream.source_remote`). | `git remote add <name> <url>`. |
| `role-conflict` | any cascade command (`loadConfig`) | `downstream.source_remote` is set **and** any local (non-remote-tracking) `channel/*`, `skill/*`, `module/*`, or `edition/*` branch exists — the source-repo composition signature. | Remove `downstream.source_remote` from `.cascade/config.yaml` if this is a source repo, or delete the stray composition branches if this is a downstream repo that shouldn't have them. |
| `stale-merge` | `cascade propagate` (pre-flight, mid-run step 1) | `MERGE_HEAD` present but not an ancestor of the current plan's next pending hop's source tip. | Abort the merge (`git merge --abort`) and re-run; the prior-session context is no longer valid. |
| `merge-in-progress` | `cascade propagate` (mid-run) | `MERGE_HEAD` present and reachable from the current hop's source tip (ancestor check, not SHA equality — fetch between runs may advance the source without invalidating the in-progress merge). | Finish (`git commit`) or abort (`git merge --abort`), then re-run. |
| `fetch-failed` | `cascade propagate` (pre-flight) | Pre-flight setup fetch failed (network error, auth failure, partial fetch). | Resolve the network / credential issue; re-run, or pass `--no-fetch` if the plan against the local view is acceptable. |
| `merge-conflict` | `cascade propagate` (mid-run) | Git reported a conflict during the merge. | Resolve, `git commit`, re-run. |
| `prefix-mismatch` | `cascade propagate` (mid-run), `cascade tag` (editions) | Sources being merged disagree on `A.B.C` and no `parent_branch` declared. | Advance the sources to the same prefix (run propagation from `core` through each source first), or declare `.cascade/parent_branch` on the target. |
| `tag-version-mismatch` | `cascade propagate` (mid-run), `writeTag()` backstop | Predicted tag name already exists at a different commit or different version. | Delete the local-only stale tag (`git tag -d <tag>`) or follow-up-bump. |
| `source-tag-missing` | `cascade propagate` (mid-run), `cascade tag` (first-bump) | First-bump baseline walk found no source tag reachable from the merge-base. | `cascade tag <branch> --seed <A.B.C.D>`; or create the missing source tag locally if it represents a known release; or re-run without `--no-fetch` if the expected tag lives upstream. |
| `unsupported-snapshot-version` | `cascade snapshot`, `cascade propagate` (downstream, when reading edition tags) | Tag's `schema_version` exceeds the CLI's max. | Update the `cascade/` submodule in the consumer repo. |
